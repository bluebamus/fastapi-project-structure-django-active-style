1왜 "세션 관리"가 따로 다룰 만큼 중요한가

데이터베이스 세션은 빌린 자원입니다. 커넥션 풀에서 연결 하나를 빌려와 쓰고, 끝나면 반드시 돌려줘야 합니다. 돌려주지 않으면 풀이 고갈되어 모든 요청이 멈춥니다.

커넥션 풀 (pool_size=20, max_overflow=20 → 최대 40)
[ ● ● ● ● ● ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ]
● 사용 중(빌려감) · ○ 대기(반납됨) — 안 돌려주면 ●가 쌓여 고갈 → 장애

그래서 "빌렸으면 무슨 일이 있어도(예외가 나도) 닫는다"가 철칙입니다. 파이썬에는 이 열고 → 쓰고 → 반드시 닫고를 자동 보장하는 두 장치가 있고, 본 프로젝트는 상황에 따라 둘 다 씁니다.

핵심은 점선 — 세션을 어떻게 만들었는지(yield냐 async with냐)와 무관하게, 그 안에서 일하는 Repository는 완전히 동일합니다. Repository는 세션을 주입받아 쓸 뿐, 세션의 '출생 신고서'를 보지 않습니다.

3패턴 A · AsyncGenerator (get_session)

3.1 실제 코드 — session.py:133

from typing import AsyncGenerator

async def get_session() -> AsyncGenerator[AsyncSession, None]:
    async with AsyncSessionLocal() as session:
        try:
            yield session          # ← yield 가 있으니 '비동기 제너레이터'
        except Exception as e:
            await session.rollback()
            raise e
        # async with 블록이 끝나며 세션 자동 close

3.2 왜 타입이 AsyncGenerator[AsyncSession, None] 인가

함수 본문에 yield가 하나라도 있으면, 그 함수는 "값을 return하는 함수"가 아니라 "값을 차례로 내보내는 제너레이터"가 됩니다. 비동기 함수에서 yield를 쓰면 정체는 AsyncGenerator이고, 타입 힌트도 이를 따라야 합니다.

AsyncGenerator[ YieldType , SendType ]
                    │           └ .asend() 로 받는 타입 (안 쓰면 None)
                    └ yield 로 내보내는 값 = AsyncSession
한 줄 요약 — yield를 쓰는 순간 AsyncGenerator 타입은 선택이 아니라 필수입니다.

3.3 FastAPI는 이 제너레이터를 어떻게 운전하나

sequenceDiagram participant FA as FastAPI participant GS as get_session() participant H as Route 핸들러 FA->>GS: 요청 도착, yield 직전까지 실행(세션 생성) GS-->>FA: session 을 yield FA->>H: session 주입 alt 정상 종료 H-->>FA: 응답 FA->>GS: 제너레이터 재개 → async with 종료(close) else 예외 발생 H-->>GS: 예외 전파 GS->>GS: except → rollback → 재전파 GS->>GS: 그 뒤 close end

요청 1건 = 세션 1개 = 트랜잭션 경계 1개. 암묵적이고 간결합니다.

# router.py
@router.get("/logs")
async def get_logs(session: AsyncSession = Depends(get_session)):
    repo = UserAccessLogRepository(session)
    return await repo.get_recent_logs()

4패턴 B · Context Manager (BaseUnitOfWork)

본 프로젝트의 BaseUnitOfWork는 초기 스케치보다 한 단계 진화해 있습니다. 세션 주입/자동생성, 일반/백그라운드 풀, Repository 자동 초기화를 한 클래스가 모두 담당합니다 (과거의 Background…UnitOfWork 쌍둥이 클래스는 제거됨).

4.1 실제 코드 — unit_of_work.py:41

class BaseUnitOfWork:
    repositories: dict[str, type] = {}        # 선언만 하면 자동 초기화

    def __init__(self, session: AsyncSession | None = None, *, background: bool = False):
        self._session = session
        self._owns_session = session is None  # 내가 만들었으면 내가 닫는다
        self._background = background          # 백그라운드 풀 사용 여부

    async def __aenter__(self) -> Self:
        if self._owns_session:                # 주입 안 받았으면 직접 생성
            factory = BackgroundSessionLocal if self._background else AsyncSessionLocal
            self._session = factory()
        for attr, repo_cls in self.repositories.items():  # Repository 자동 결선
            setattr(self, attr, repo_cls(self._session))
        return self

    async def __aexit__(self, exc_type, exc_val, exc_tb) -> None:
        if exc_type is not None:
            await self.rollback()             # 예외 시 자동 롤백
        if self._owns_session and self._session:
            await self._session.close()       # 내가 만든 세션만 내가 닫는다

도메인 하위 클래스는 Repository 목록만 선언하면 됩니다.

# app/domains/home/unit_of_work.py
class HomeUnitOfWork(BaseUnitOfWork):
    user_access_logs: UserAccessLogRepository
    repositories = {"user_access_logs": UserAccessLogRepository}

4.2 yield가 없으므로 AsyncGenerator도 없다

패턴 B는 yield를 한 번도 쓰지 않습니다. 대신 __aenter__(진입)와 __aexit__(종료) 매직 메서드가 "열고/닫고"를 담당합니다. 따라서 from typing import AsyncGenerator import 자체가 불필요합니다.

sequenceDiagram participant Caller as 호출부 (async with) participant UoW as HomeUnitOfWork participant Repo as Repository participant DB as DB 세션/풀 Caller->>UoW: async with HomeUnitOfWork() UoW->>DB: __aenter__ → 세션 생성(owns=True) UoW->>Repo: repositories 맵으로 자동 초기화 Caller->>Repo: uow.user_access_logs.create(data) Caller->>UoW: uow.commit() UoW->>DB: session.commit() Note over UoW,DB: 블록 종료 UoW->>DB: __aexit__ → 예외면 rollback, owns면 close

4.3 사용 예

# 서비스 / 백그라운드 태스크
async def save_access_log(data: dict):
    async with HomeUnitOfWork() as uow:        # 세션 자동 생성
        await uow.user_access_logs.create(data)
        await uow.commit()                     # 트랜잭션 경계가 '명시적'

# 백그라운드 전용 풀 — 클래스 교체 없이 플래그만
async def save_in_background(data: dict):
    async with HomeUnitOfWork(background=True) as uow:
        await uow.user_access_logs.create(data)
        await uow.commit()
세션 소유권(ownership) — UoW가 외부에서 세션을 주입받으면(_owns_session=False) 닫지 않습니다. FastAPI가 준 세션을 UoW가 멋대로 닫아버리는 사고를 막는 안전장치입니다. 덕분에 패턴 A의 세션을 패턴 B로 그대로 넘겨 재사용할 수 있습니다 — async with HomeUnitOfWork(session) as uow:

8결론

  1. AsyncGenerator는 yield를 쓰는 비동기 함수의 타입 힌트로 필수 — FastAPI Depends와 가장 잘 맞습니다.
  2. 피하려면 __aenter__/__aexit__를 구현한 클래스 기반 컨텍스트 매니저를 쓰면 됩니다 — 본 프로젝트의 BaseUnitOfWork가 그것.
  3. 두 패턴은 경쟁이 아니라 분업입니다. 요청 단위 간결함은 A가, 명시적 트랜잭션·다중 Repository·백그라운드 분리는 B가 담당.
  4. 어느 쪽이든 Repository는 세션을 주입받아 동일하게 동작하므로, 패턴 선택이 데이터 접근 코드에 영향을 주지 않습니다.
