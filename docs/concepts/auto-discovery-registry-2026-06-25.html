이 문서의 범위. 이 문서는 feature 브랜치 의 자동 발견(Auto-Discovery) 설계를 다룹니다. 앱을 어디에도 "등록"하지 않고, app/domains/ 디렉터리에 폴더가 존재하는 것만으로 자동 활성화하는 "convention over configuration" 방식입니다. 명시적 목록(INSTALLED_APPS) 방식은 자매 문서 app-registration-installed-apps(main 브랜치)가 다룹니다. 두 브랜치는 앱 목록을 얻는 방법(discover())만 다르고, 그 뒤의 결선·서비스·DI 구조는 완전히 동일합니다.

feature 브랜치 설계 — 컨벤션 기반 자동 발견

자동 발견(Auto-Discovery)은 개발자가 "이 앱을 켜라"라고 어디에도 적지 않아도, 부팅 시점에 프로그램이 디렉터리를 뒤져 앱을 스스로 찾아내는 방식입니다. 이 프로젝트의 feature 브랜치는 app/domains/ 바로 아래의 하위 패키지를 모두 앱으로 간주하고, 이름 기반 컨벤션으로 라우터·모델·Admin을 자동 결선합니다. 새 앱을 추가할 때 중앙 파일을 한 줄도 고치지 않습니다.

핵심을 한 줄로 요약하면 이렇습니다. "앱을 켜는 일(등록)"과 "켜진 앱을 연결하는 일(결선)"을 모두 규칙이 자동으로 한다. 사람은 약속된 디렉터리 구조와 이름만 지키면 됩니다. 이 문서는 그 자동 발견이 어떻게 동작하는지, 그리고 그 위에서 라우터·서비스·DI가 어떻게 조립되는지를 끝까지 따라갑니다.

결론 요약. AppRegistry.discover() 가 부팅 시 app/domains/* 직계 하위 패키지를 스캔(언더스코어 디렉터리 제외, 알파벳순)하여 각 앱을 발견합니다. 앱별 config.py 나 AppConfig 선언은 없습니다(gen-2). 발견 이후 라우터는 /api/v1/<name> 에 마운트되고, 각 요청은 View → Service → Repository → DB 계층을 타며, 트랜잭션 경계는 의존성(dependency) 이 yield 후 커밋으로 책임집니다(UnitOfWork 제거).

0. 두 세대 — 무엇이 "gen-2"인가

이 프로젝트의 자동 발견은 두 세대를 거쳤습니다. 두 세대 모두 "자동"이지만, 앱이 자신을 어떻게 드러내는가가 다릅니다.

세대앱의 자기 선언발견 방식상태
gen-1 (이전)앱마다 config.py 에 AppConfig 하위 클래스config.py 모듈을 스캔해 AppConfig 수집대체됨
gen-2 (현행)선언 없음 — 디렉터리 구조가 곧 선언app/domains/* 패키지 자체를 스캔현행

gen-1은 Django의 AppConfig 를 가장 충실히 흉내 낸 형태였습니다. 각 앱이 config.py 에 "나는 이런 앱이고 내 라우터는 여기 있다"라고 적었고, 레지스트리가 그 선언을 모았습니다. gen-2는 한 발 더 나아가 그 선언마저 없앴습니다. "디렉터리가 있으면 그게 앱"이라는 컨벤션으로 바꾼 것입니다. 그 결과 앱마다 반복되던 config.py 보일러플레이트가 사라졌습니다.

왜 gen-2로 갔나. main 브랜치의 수동 등록은 INSTALLED_APPS = [이름] 입니다 — 이름만 있습니다. 그 이름만으로 라우터를 연결하려면 결국 이름 기반 컨벤션(<name>_router 등)이 필요합니다. 그런데 결선을 이름 컨벤션으로 통일하고 나면, gen-1의 config.py 선언은 더 이상 필요가 없어집니다. feature는 그 컨벤션을 그대로 쓰되 "이름 목록"을 스캔으로 채울 뿐입니다. 그래서 두 브랜치가 같은 결선 코드를 공유할 수 있게 되었습니다(이것이 gen-2의 진짜 동기).

그림 1. 자동 발견 허브 — 입력이 "명시적 목록"이 아니라 "디렉터리 자체"다. 그 외 결선 엔진과 조립 결과는 main과 동일하다.

main 문서의 그림 1과 비교하면, 왼쪽 상자만 INSTALLED_APPS 리스트에서 app/domains/ 디렉터리로 바뀌었습니다. 오른쪽(결선 엔진·조립 결과)은 글자 그대로 같습니다. 즉 자동/수동의 차이는 "앱 목록을 어디서 얻느냐" 한 곳에 격리되어 있습니다.

2. 앱의 해부학 — 디렉터리가 곧 선언

자동 발견에서 디렉터리 구조는 단순한 정리 정돈이 아니라 "이 앱은 이렇게 생겼다"는 선언 그 자체입니다. 레지스트리는 약속된 위치·이름을 보고 구성요소를 찾습니다. 아래는 blog 앱의 표준 레이아웃입니다(구조는 main과 동일하며, 차이는 config.py 가 없다는 점뿐입니다).

"없으면?" 열이 자동 발견의 관용성을 보여줍니다. 라우터가 없는 디렉터리도 "앱"으로는 발견되지만, 마운트할 라우터가 없으니 경고 로그를 남기고 조용히 건너뜁니다. 모델이 없으면 테이블 등록을 건너뛰고, Admin이 없으면 Admin 뷰가 없을 뿐입니다. 즉 각 구성요소는 독립적으로 "있으면 결선, 없으면 무시" 됩니다. 덕분에 최소한의 앱(라우터만 있는 앱)도 문제없이 동작합니다.

3. 앱을 만드는 절차 — 폴더만 만들면 끝

자동 발견에서 앱 추가는 단 하나의 본질적 행위로 줄어듭니다. 약속된 구조의 폴더를 만드는 것. 별도의 "등록" 단계가 없습니다.

그림 3. 앱 생성 파이프라인(feature) — main 그림 3과 비교해 "③ 목록 등록" 단계가 통째로 사라졌다. 폴더가 곧 등록이다.

main 브랜치의 파이프라인(그림 3)에는 "③ INSTALLED_APPS 에 추가"라는 노란 단계가 있었습니다. feature에는 그 단계가 없습니다. 스캐폴더가 만든 폴더가 그 자리에 있는 한, 다음 부팅에서 자동으로 발견됩니다. 스캐폴더는 gen-2 컨벤션을 따르므로 config.py 를 생성하지 않으며, 워커 옵션(--with-worker)도 없습니다(워커는 중앙 app/celery).

자동의 그림자. 편한 만큼 위험도 있습니다. app/domains/ 아래에 실험용·미완성 폴더를 두면 의도치 않게 함께 켜집니다. 잠시 끄고 싶다면 폴더 이름을 언더스코어로 시작하게 바꾸거나(_draft_blog), app/domains 밖으로 옮기면 됩니다(스캔 제외). 명시적 on/off 게이트가 필요하면 main의 수동 방식이 더 안전합니다.

4. 발견 메커니즘 — discover()의 내부

등록 메커니즘의 구성요소는 main과 동일합니다 — 데이터 홀더 AppModule 과 엔진 AppRegistry. 차이는 오직 discover() 의 본문입니다.

그림 4. 레지스트리 클래스 다이어그램 — AppModule 과 결선 메서드는 main과 byte-identical. 분기점은 discover() 한 곳뿐이다.

feature 브랜치의 discover() 는 pkgutil.iter_modules 로 app/domains 직계 하위 패키지를 훑습니다. 언더스코어로 시작하는 것은 제외하고, 이름 알파벳순으로 정렬한 뒤 각 앱 패키지를 import 합니다(부수효과).

# app/core/registry.py — feature 브랜치
def discover(self, package: str = DOMAINS_PACKAGE) -> list[AppModule]:
    root = importlib.import_module(package)
    names: list[str] = []
    for info in pkgutil.iter_modules(root.__path__):
        if not info.ispkg or info.name.startswith("_"):
            continue                                  # 패키지가 아니거나 _시작이면 제외
        names.append(info.name)
    apps = [AppModule(name=n, package=f"{package}.{n}") for n in sorted(names)]
    for app in apps:
        importlib.import_module(app.package)          # __init__.py 부수효과(예: home sink)
    self._apps = apps
    return apps

그림 5. discover() 시퀀스(feature) — 디렉터리를 나열하고, 비-패키지·언더스코어를 걸러 정렬한 뒤, 각 앱 패키지를 import해 부수효과를 깨운다.
단계처리비고
1. 나열pkgutil.iter_modules(domains.__path__)직계 하위만(재귀 아님)
2. 필터패키지가 아니거나 _ 시작이면 제외__pycache__ 등 자동 제외
3. 정렬sorted(names)알파벳순(결정적)
4. 매핑·부수효과AppModule 생성 + 패키지 import__init__.py 실행

main의 discover()(목록 읽기)와 이 코드(디렉터리 스캔)를 나란히 보면, 마지막 두 줄(AppModule 생성, 패키지 import, self._apps 보관)이 동일합니다. 바뀐 것은 "이름 목록을 어떻게 만드느냐"뿐입니다. main은 리스트에서 읽고, feature는 디스크를 훑습니다. 그 이후의 결선은 완전히 같은 코드를 탑니다.

5. 라우터 등록 흐름 — 경로는 어떻게 합성되나

# app/core/registry.py (결선 — 두 브랜치 공유, 완전히 동일)
def install_routers(self, app) -> int:
    count = 0
    for module in self._apps:
        router = module.load_router()          # _router 를 컨벤션으로 찾음
        if router is None:
            logger.warning("앱 '%s' 에 %s 라우터가 없어 건너뜀", module.name, module.router_attr)
            continue
        app.include_router(router, prefix=module.prefix)   # prefix = "/api"
        count += 1
    return count

# app/domains/blog/services/blog_service.py (main과 동일)
class BlogService(BaseService):
    def __init__(self, session: AsyncSession) -> None:
        super().__init__(session)
        self.repository = PostRepository(session)

    async def get_post(self, post_id: str) -> Post:
        self.log.debug("게시글 조회: id=%s", post_id)
        post = await self.repository.get_by_id(post_id)
        if post is None:
            raise PostNotFoundException(detail={"id": post_id})
        return post

이 계층 구조가 자동/수동 어느 쪽이든 동일하다는 점이 중요합니다. 앱을 어떻게 발견하든, 일단 발견된 뒤의 내부 동작(요청이 뷰→서비스→리포지토리를 타는 흐름)은 한 가지뿐입니다. 발견 방식은 "현관문"의 차이일 뿐, "집 내부 구조"는 같습니다.

7. DI 결합·주입 — 트랜잭션 경계는 의존성이 책임진다

UnitOfWork를 제거하고 그 역할을 의존성으로 옮긴 설계도 main과 동일합니다. 뷰는 "비즈니스 흐름 함수"(의존성)를 주입받고, 그 의존성이 서비스를 조립해 건넨 뒤 성공 시 커밋합니다. 

# app/domains/blog/dependencies/blog_dependencies.py
async def get_blog_service(
    session: AsyncSession = Depends(get_session),
) -> AsyncGenerator[BlogService, None]:
    service = BlogService(session)   # ① 세션으로 서비스 조립
    yield service                    # ② 뷰에 주입
    await session.commit()           # ③ 성공 시 커밋 (트랜잭션 경계)

자동 발견과 직접 맞닿는 부분은 마지막 행입니다. home 앱의 접속 로그 sink는 요청 밖에서 동작하는데, 그 등록은 home/__init__.py 의 import 시점에 일어납니다. 그리고 그 import를 트리거하는 것이 바로 discover() 의 "각 앱 패키지 import" 단계입니다. 즉 자동 발견이 앱을 찾는 행위가, 그 앱의 초기화(sink 등록)까지 자연스럽게 함께 일으키는 구조입니다.

10. 새 앱을 추가하는 실전 가이드

order 앱을 추가한다고 합시다. feature 브랜치에서는 더 짧습니다.

  1. 골격 생성: python -m scripts.new_app order → app/domains/order/ 컨벤션 골격.
  2. 본체 작성: models/ · schemas/ · repositories/ · services/ · dependencies/ · api/routers/v1/order.py. router.py 가 v1을 /v1/order 로 include.
  3. 등록 단계 없음: 폴더가 app/domains/ 아래에 있는 것으로 충분하다. 별도 목록 편집이 없다.
  4. 확인: 서버 재시작 → 자동 발견 → /api/v1/order/... 마운트, DEBUG면 테이블 생성.
이름 컨벤션 체크리스트(main과 동일). 라우터 변수 order_router, 모델은 models/ 패키지, Admin은 admin.py 의 admin_views, import-time 초기화는 __init__.py. 다만 feature에서는 INSTALLED_APPS 추가 단계가 없다는 점만 기억하면 됩니다.

11. feature(자동) ↔ main(수동) 비교

마지막으로 두 브랜치를 나란히 둡니다. 차이는 단 하나, "앱 목록을 어떻게 얻는가"뿐이며 그 외 모든 것은 동일합니다.

항목feature (이 문서)main
앱 목록 출처app/domains/* 디렉터리 스캔config.INSTALLED_APPS (사람이 작성)
활성화 행위폴더 생성만으로 자동목록에 이름 추가(명시적)
로드 순서알파벳순리스트 순서(명시 제어)
"디스크 존재 = 활성"예(있으면 켜짐)아니오(목록이 게이트)
앱 끄기폴더를 옮기거나 _ 접두목록에서 한 줄 삭제
discover() 구현pkgutil 스캔INSTALLED_APPS 읽기
결선 로직 / AppModule완전히 동일 (byte-identical)
계층·DI·서비스·예외완전히 동일

feature는 "속도와 무설정(zero-config)"을 택한 브랜치입니다. 새 앱을 만들면 즉시 켜지고, 중앙 파일을 만질 일이 없습니다. 대신 게이트가 없으므로, 실험용 폴더가 우연히 켜지는 것을 막으려면 명명 규칙(언더스코어)이나 위치 이동에 의존해야 합니다. 명시적 통제가 중요한 환경이라면 main의 수동 방식이 더 맞습니다.

한눈에. 두 문서를 함께 읽으면 이 프로젝트의 설계 의도가 분명해집니다. "자동이냐 수동이냐"는 철학의 차이처럼 보이지만, 코드 차원에서는 discover() 메서드 하나로 격리되어 있습니다. 그래서 같은 코드베이스 위에서 두 철학을 모두 시연할 수 있습니다. 수동 방식의 상세는 자매 문서 app-registration-installed-apps 를 보세요.
