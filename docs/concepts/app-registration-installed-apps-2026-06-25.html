main 브랜치 설계 — INSTALLED_APPS 수동 앱 등록

Django는 settings.INSTALLED_APPS 라는 하나의 리스트로 "어떤 앱이 활성인가"를 선언합니다. FastAPI에는 이런 내장 개념이 없습니다. 이 프로젝트의 main 브랜치는 그 개념을 그대로 재현합니다. 프로젝트 루트 config.py 안의 INSTALLED_APPS 리스트가 단일 선언 지점(SSOT)이고, 부팅 시 AppRegistry 가 이 목록을 읽어 각 앱의 라우터·모델·Admin을 이름 기반 컨벤션으로 자동 결선합니다.

핵심을 한 줄로 요약하면 이렇습니다. "앱을 켜는 일(등록)"은 사람이 목록에 한 줄 적어 명시적으로 하고, "켜진 앱을 연결하는 일(결선)"은 규칙이 자동으로 한다. 이 문서는 그 두 단계가 어떻게 맞물리는지, 그리고 그 위에서 라우터·서비스·DI가 어떻게 조립되는지를 끝까지 따라갑니다.

결론 요약. config.INSTALLED_APPS = ["home","blog","reply","sns","user"] 다섯 앱이 등록되어 있고, 부팅 시 AppRegistry.discover() 가 이 목록을 순서 그대로 읽어 각 앱을 app/domains/<name> 로 매핑합니다. 이후 라우터는 /api/v1/<name> 경로에 마운트되고, 각 요청은 View → Service → Repository → DB 계층을 타며, 트랜잭션 경계는 의존성(dependency) 이 yield 후 커밋으로 책임집니다(과거의 UnitOfWork는 제거됨).

1. 큰 그림 — 수동 등록의 허브 구조

먼저 전체 지형도부터 봅니다. 가운데에 config.INSTALLED_APPS 라는 단일 목록이 있고, 부팅 시 AppRegistry 가 이 목록을 읽어 각 도메인 앱을 연결합니다. "목록에 이름이 있으면 활성, 없으면 디스크에 코드가 있어도 비활성"이라는 Django의 두 가지 핵심 성질이 그대로 성립합니다.

그림 1. 허브 구조 — INSTALLED_APPS(선언)가 AppRegistry(결선 엔진)를 거쳐 도메인 앱과 조립 결과를 잇는다. 선언과 결선이 분리되어 있는 것이 핵심이다.

이 구조에서 "선언"과 "결선"이 분리되어 있다는 점을 눈여겨봐야 합니다. 사람이 손대는 곳은 오직 왼쪽의 INSTALLED_APPS 리스트 하나뿐입니다. 그 오른쪽(레지스트리·결선)은 어떤 앱이 추가되든 코드를 바꿀 필요가 없습니다. 바로 이 "결선 엔진"이 feature 브랜치와 한 글자도 다르지 않게 공유되는 부분입니다. 두 브랜치의 차이는 왼쪽 상자(목록을 사람이 적느냐, 스캔으로 채우느냐)뿐입니다.

표에서 가장 실무적으로 중요한 행은 "디스크 존재 ≠ 활성"입니다. app/domains/ 아래에 폴더를 만들어 두기만 해서는 아무 일도 일어나지 않습니다. 반드시 INSTALLED_APPS 에 이름을 적어야 비로소 라우터가 마운트되고 테이블이 생성됩니다. 이것이 수동 등록의 본질이자, 의도된 안전장치입니다(원치 않는 앱이 우연히 켜지지 않는다).

표의 "레지스트리가 보는가?" 열이 컨벤션의 핵심입니다. 레지스트리는 오직 네 가지 약속된 이름만 봅니다 — 앱 패키지(__init__.py), 라우터(<name>_router), 모델 패키지(models), Admin(admin_views). 나머지(스키마·서비스·리포지토리·의존성)는 레지스트리가 전혀 알 필요가 없습니다. 이들은 앱 내부에서 서로를 import로 연결할 뿐이므로, 자유롭게 추가·분리해도 등록 메커니즘에 영향을 주지 않습니다. 컨벤션이 "최소한"이어서 유연한 것입니다.

3. 앱을 만드는 절차 — 스캐폴딩 → 등록

새 앱을 만드는 일은 두 단계로 나뉩니다. (1) 골격을 생성하고, (2) 목록에 등록합니다. main 브랜치에서는 두 번째 단계가 사람의 명시적 행위입니다(이것이 "수동"의 의미).

그림 3. 앱 생성 파이프라인 — 노란 ③단계(목록 등록)만 사람의 명시적 행위다. 나머지는 골격 생성과 결선이 자동이다.

스캐폴더(scripts/new_app.py)는 컨벤션에 맞는 빈 골격을 만들어 줍니다. 라우터 집약 파일에는 이미 blog_router = APIRouter() 가 들어 있고, 의존성 폴더와 모델·스키마·서비스·리포지토리 디렉터리가 __init__.py 와 함께 준비됩니다. 스캐폴더는 더 이상 config.py 를 만들지 않습니다(gen-2 컨벤션). 과거에 있던 --with-worker 옵션도 제거되었습니다(워커는 중앙 app/celery 로 일원화).

잊기 쉬운 한 줄. 골격을 만들고 본체까지 다 작성해도, INSTALLED_APPS 에 이름을 추가하지 않으면 앱은 조용히 비활성 상태로 남습니다. 라우트가 안 보이거나 테이블이 안 생긴다면 가장 먼저 이 목록을 확인하세요. (자동 발견 브랜치에는 이 단계가 없습니다 — 트레이드오프는 §11.)

4. 등록 메커니즘 — INSTALLED_APPS → AppRegistry → AppModule

이제 등록의 내부를 봅니다. 두 개의 작은 구성요소가 전부입니다 — 데이터 홀더 AppModule 과 엔진 AppRegistry.

그림 4. 레지스트리 클래스 다이어그램 — AppModule 은 "이름 하나로 라우터/모델/Admin을 찾는 법"을 안다. AppRegistry 는 목록을 만들고(discover) 결선한다(install_*).

AppModule 은 불변(frozen) 데이터클래스로, 앱 이름과 패키지 경로만 가집니다. 그리고 그 이름으로부터 컨벤션에 따라 구성요소를 찾는 법을 압니다. 예를 들어 name="blog" 이면 router_attr 는 "blog_router" 이고, load_router() 는 app.domains.blog.api.routers.router 모듈에서 그 이름의 변수를 가져옵니다. 모듈이나 변수가 없으면 조용히 None 을 반환합니다(없는 것은 그냥 건너뜀).

main 브랜치의 discover() 는 다음과 같이 동작합니다. 목록을 순서대로 읽어 각 이름을 AppModule 로 만들고, 각 앱 패키지를 import 합니다(부수효과 실행).

# app/core/registry.py — main 브랜치
def discover(self, package: str = DOMAINS_PACKAGE) -> list[AppModule]:
    from config import INSTALLED_APPS
    apps = [AppModule(name=n, package=f"{package}.{n}") for n in INSTALLED_APPS]
    for app in apps:
        importlib.import_module(app.package)   # __init__.py 부수효과(예: home sink)
    self._apps = apps
    return apps

3단계의 "부수효과"가 중요합니다. home 앱은 접속 로그를 DB에 쓰는 sink를 미들웨어에 등록해야 하는데, 이 등록을 app/domains/home/__init__.py 의 import 시점에 수행합니다. 레지스트리가 앱 패키지를 import하는 순간 그 코드가 실행되어 sink가 자동으로 연결됩니다. 즉 "앱을 등록하면 그 앱의 초기화도 함께 일어난다"는 점이 이 import 한 줄에 담겨 있습니다.

그림 9. 서비스·리포지토리 클래스 다이어그램(blog 예) — 서비스는 BaseService(세션·로깅)를, 리포지토리는 BaseRepository(CRUD)를 상속한다. 서비스가 리포지토리를 합성한다. 
# app/domains/blog/services/blog_service.py
class BlogService(BaseService):
    def __init__(self, session: AsyncSession) -> None:
        super().__init__(session)
        self.repository = PostRepository(session)   # 세션으로 리포지토리 구성

    async def get_post(self, post_id: str) -> Post:
        self.log.debug("게시글 조회: id=%s", post_id)   # LoggerMixin → self.log
        post = await self.repository.get_by_id(post_id)
        if post is None:
            raise PostNotFoundException(detail={"id": post_id})  # 도메인 예외
        return post
계층기반 클래스아는 것모르는 것
View—HTTP 파라미터·응답 모델·의존성SQL, 트랜잭션, 비즈니스 규칙
Dependency—서비스 생성법·커밋 시점비즈니스 로직 내부
ServiceBaseService(LoggerMixin)비즈니스 규칙·도메인 예외·세션HTTP, 라우팅
RepositoryBaseRepository[Model]SQLAlchemy 쿼리·CRUD비즈니스 의미·HTTP

이 분리의 실익은 "변경의 국소성"입니다. URL을 바꾸면 뷰만, 비즈니스 규칙을 바꾸면 서비스만, 쿼리를 최적화하면 리포지토리만 손대면 됩니다. 특히 서비스가 BaseService 를 통해 LoggerMixin 을 상속하므로, 모든 서비스 메서드는 self.log 로 클래스 이름이 자동으로 찍히는 로거를 별도 설정 없이 사용합니다.

7. DI 결합·주입 — 트랜잭션 경계는 의존성이 책임진다

이 프로젝트의 가장 중요한 설계 결정은 UnitOfWork를 제거하고, 그 역할을 FastAPI의 의존성(dependency)으로 옮긴 것입니다. 뷰는 "비즈니스 흐름을 정의하는 함수"(=의존성)를 주입받고, 그 의존성이 서비스를 만들어 건네준 뒤 요청이 성공하면 커밋합니다. 

# app/domains/blog/dependencies/blog_dependencies.py
async def get_blog_service(
    session: AsyncSession = Depends(get_session),
) -> AsyncGenerator[BlogService, None]:
    service = BlogService(session)   # ① 세션으로 서비스 조립
    yield service                    # ② 뷰에 주입 (여기서 뷰 실행)
    await session.commit()           # ③ 뷰가 성공 반환하면 커밋 (트랜잭션 경계)

# app/domains/blog/api/routers/v1/blog.py
@router.post("/posts", response_model=PostResponse, status_code=201)
async def create_post(
    payload: PostCreate,
    service: BlogService = Depends(get_blog_service),   # 주입
) -> PostResponse:
    post = await service.create_post(payload)   # HTTP만; 로직은 서비스
    return PostResponse.model_validate(post)

그림 10. 요청 라이프사이클 — 의존성이 세션→서비스를 조립해 주입하고, 뷰가 성공 반환한 직후 커밋한다. 예외가 나면 커밋 줄에 도달하지 못하고 get_session 이 롤백한다.

이 한 장의 시퀀스에 트랜잭션 의미론이 모두 담겨 있습니다. yield 이전은 "준비"(세션·서비스 조립), yield 이후는 "마무리"(커밋)입니다. 뷰 본문에서 도메인 예외(PostNotFoundException 등)가 발생하면 yield 다음 줄(commit())에 도달하지 못하고, get_session 의 except 절이 롤백을 수행합니다. 그래서 "성공하면 정확히 한 번 커밋, 실패하면 자동 롤백"이 코드 흐름만으로 보장됩니다 — 별도의 UnitOfWork 객체가 필요 없습니다.

관심사과거(UnitOfWork)현재(의존성 DI)
세션 수명UoW가 열고 닫음get_session(요청 스코프 DI)
트랜잭션 경계async with uow:의존성의 yield 후 commit()
리포지토리 결선UoW가 보유서비스가 세션으로 생성
요청 밖 작업UoW 재사용background_session() 컨텍스트

마지막 행의 "요청 밖 작업"은 Celery 태스크나 접속 로그 sink처럼 HTTP 요청 스코프가 없는 곳을 말합니다. 이때는 의존성을 쓸 수 없으므로 background_session() 이라는 비동기 컨텍스트 매니저가 같은 역할(세션 제공 + 예외 시 롤백)을 합니다. 커밋만 호출자가 명시적으로 합니다.

10. 새 앱을 추가하는 실전 가이드

예를 들어 order 앱을 추가한다고 합시다. main 브랜치에서의 절차는 다음과 같습니다.

  1. 골격 생성: python -m scripts.new_app order → app/domains/order/ 컨벤션 골격이 생긴다.
  2. 본체 작성: models/ 에 ORM, schemas/ 에 Pydantic, repositories/ · services/ · dependencies/ · api/routers/v1/order.py 작성. router.py 가 v1을 /v1/order 로 include하도록 한다.
  3. 등록 (수동·필수): config.INSTALLED_APPS 에 "order" 를 추가한다. 위치(순서)가 로드 순서가 된다.
  4. 확인: 서버 재시작 후 /api/v1/order/... 가 마운트되고, DEBUG면 테이블이 생성된다.
이름 컨벤션 체크리스트. 라우터 변수는 반드시 order_router(=<name>_router), 모델은 models/ 패키지에서 import 가능, Admin을 쓰면 admin.py 에 admin_views = [...], import-time 초기화가 필요하면 __init__.py 에 둔다. 이 네 가지 컨벤션만 지키면 레지스트리가 나머지를 알아서 결선합니다.

11. main(수동) ↔ feature(자동) 비교

마지막으로 두 브랜치를 나란히 둡니다. 반복하지만 차이는 단 하나, "앱 목록을 어떻게 얻는가" 뿐입니다. 그 외 모든 것(결선·계층·DI·서비스·컨벤션)은 동일합니다.

항목main (이 문서)feature
앱 목록 출처config.INSTALLED_APPS (사람이 작성)app/domains/* 디렉터리 스캔
활성화 행위목록에 이름 추가(명시적)폴더 생성만으로 자동
로드 순서리스트 순서(명시 제어)알파벳순
"디스크 존재 = 활성"아니오(목록이 게이트)예(있으면 켜짐)
discover() 구현INSTALLED_APPS 읽기pkgutil 스캔
결선 로직 / AppModule완전히 동일 (byte-identical)
계층·DI·서비스·예외완전히 동일

표의 마지막 두 행이 이 프로젝트의 설계 미학입니다. "자동이냐 수동이냐"라는 큰 차이처럼 보이는 선택이, 실제로는 discover() 메서드 하나의 본문 차이로 격리되어 있습니다. 덕분에 두 브랜치는 같은 코드베이스를 공유하면서 서로 다른 등록 철학을 시연할 수 있습니다. 자동 발견의 내부가 궁금하다면 자매 문서 auto-discovery-registry 를 이어서 보세요.

한눈에. main은 "안전과 명시성"을 택한 브랜치입니다. 목록이 게이트 역할을 하므로 의도치 않은 앱이 켜지지 않고, 로드 순서를 사람이 통제합니다. 대신 새 앱마다 목록에 한 줄을 더하는 수고가 있습니다. 그 수고를 없앤 것이 자동 발견(feature)이며, 그 대가는 "디스크에 있으면 무조건 켜진다"는 점입니다.
